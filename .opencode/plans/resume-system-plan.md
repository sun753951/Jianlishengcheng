# 基于自然语言生成的智能简历生成系统 — 策划案

## 一、产品定位与目标用户

### 1.1 产品定位
一款运行于 HarmonyOS 手机/平板端侧的智能简历生成工具，用户通过填写个人信息与工作经历，系统利用端侧 NLG（自然语言生成）模型自动生成专业、个性化的简历文本，并支持在线编辑、评分优化与 PDF 导出。

### 1.2 核心价值主张
- **端侧隐私**：所有数据处理与模型推理在设备端完成，用户简历数据不上传云端
- **即填即得**：填写基础信息后秒级生成完整简历草稿
- **智适应优化**：根据目标岗位自动调整简历侧重点，并提供评分反馈

### 1.3 目标用户
| 用户类型 | 典型场景 | 核心需求 |
|---------|---------|---------|
| 应届毕业生 | 首次求职，缺乏简历撰写经验 | 模板引导、智能生成、语法纠错 |
| 职场跳槽人士 | 针对特定岗位定制简历 | 岗位需求分析、关键词适配、评分优化 |
| 自由职业者/多面手 | 同一份履历适配不同方向 | 多版本管理、快速切换模板 |
| HR/职业顾问 | 帮助他人优化简历 | 评分系统、结构化建议 |

## 三、用户交互流程

### 3.1 核心路径（首次创建简历）

```
启动 App → 主界面: 我的简历列表 → 点击"新建简历"
  → 填写基本信息 → 添加工作经历 → 添加项目经验
  → 选择目标岗位模板 → 选择简历样式模板
  → [确认生成?]
     → 是 → 端侧 NLG 推理生成内容 → 简历预览页
     → 否 → 返回选择
  → [满意?]
     → 是 → 导出 PDF → 保存/分享 PDF
     → 否 → 在线编辑 → 手动调整内容
         → 触发评分 → 查看优化建议 → 根据建议修改 → 回到预览
```

### 3.2 辅助流程

**简历评分与优化**：
预览页 → 点击"评分" → 系统分析四项维度 → 展示评分雷达图 + 分数 → 逐项展示优化建议 → 用户点击建议可定位到对应编辑区

**岗位匹配分析**：
简历编辑页 → "岗位匹配" → 输入或粘贴 JD 文本 → 系统高亮简历中匹配的关键词 → 列出 JD 中未覆盖的关键词

### 3.3 页面路由设计

| 页面 | 路由 | 说明 |
|------|------|------|
| 简历列表页 | `/` | 展示所有简历卡片，支持新建入口 |
| 信息编辑页 | `/editor` | 分 Tab 编辑基本信息/工作经历/项目经历 |
| 模板选择页 | `/template` | 简历风格模板选择 |
| 简历预览页 | `/preview` | 实时预览 + 评分/优化/导出入口 |
| 岗位匹配页 | `/match` | JD 输入与分析结果展示 |

## 四、非功能需求

### 4.1 性能指标

| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 冷启动时间 | ≤ 3s | App 启动到主界面可交互 |
| NLG 生成时间 | ≤ 2s/份 | 从点击"生成"到预览渲染完成（搭载 Kirin 9000 以上芯片） |
| 页面切换响应 | ≤ 200ms | 页面路由跳转完成 |
| PDF 导出时间 | ≤ 5s | 10页以内简历 |
| 模型加载时间 | ≤ 3s | 首次启动加载 ONNX 模型到内存 |
| 包体积增量 | ≤ 80MB | 模型文件 + 应用代码合计 |
| 运行时内存占用 | ≤ 300MB | 模型推理峰值 |

### 4.2 兼容性

| 维度 | 要求 |
|------|------|
| 目标系统 | HarmonyOS 4.0+（API 10+） |
| 目标设备 | 手机（推荐内存 ≥ 6GB）+ 平板 |
| 芯片兼容 | 麒麟 990+ / 骁龙 8+ 及以上（支持 NPU 加速） |
| 屏幕适配 | 折叠屏、平板横竖屏自适应布局 |
| 分辨率 | 360vp ~ 840vp 宽度自适应 |

### 4.3 安全性 & 隐私

- 所有用户数据存储于应用沙箱目录，不上传网络
- 模型文件随应用打包，不额外下载（如需模型更新，通过应用市场整包更新）
- 导出 PDF 不含任何元数据追踪信息
- 不需要联网权限（日历/联系人/位置等敏感权限均不需要）
- 如需 i18n 模型更新，支持增量下载（HSP 动态包）

### 4.4 可用性

- 支持深色模式（跟随系统）
- 支持单手操作（关键按钮在屏幕下半区）
- 文本输入区支持语音输入（调用系统听写能力）
- 误操作弹窗确认（删除经历、覆盖草稿等）

## 六、模型需求

### 6.1 模型选型建议

| 模型 | 参数规模 | ONNX 大小 | 开源许可 | 推荐度 |
|------|---------|-----------|---------|-------|
| **Chinese-GPT2-Small** | 124M | ~47MB (FP32) / ~24MB (INT8) | MIT | ⭐⭐⭐⭐⭐ |
| Chinese-GPT2-Medium | 345M | ~131MB (FP32) / ~67MB (INT8) | MIT | ⭐⭐⭐ |
| Tiny-LLaMA (Chinese) | 1.1B | 过大，不适配端侧 | — | ⭐ |
| MiniCPM-2B (量化) | 2B | ~1GB+ | 不适合端侧 | ⭐ |

**推荐方案：Chinese-GPT2-Small (124M) → ONNX INT8 量化**

### 6.2 选型理由

1. **参数规模合理**：124M 参数，INT8 量化后约 24MB，满足包体积 ≤ 80MB 限制
2. **推理速度**：在 Kirin 9000（CPU）上，INT8 量化模型推理 128 token 约 **800ms~1.2s**，满足 ≤ 2s 要求
3. **中文能力**：基于中文维基百科 + 新闻语料预训练，简历领域微调后文本质量可达 98%+ 语法正确率
4. **生态成熟**：PyTorch → ONNX 转换流程完善

### 6.3 ONNX 转换与量化方案

```
PyTorch 预训练模型 → 领域微调 → 导出 ONNX → INT8 量化 → ONNX Runtime 推理
   …↑
                  简历数据集
```

**转换步骤**：
1. 在 PC 端使用 Hugging Face Transformers 加载 `Chinese-GPT2-Small`
2. 使用简历数据集进行领域微调（LoRA + 全量微调混合策略）
3. 使用 `torch.onnx.export()` 导出为 ONNX 格式（固定序列长度 256）
4. 使用 ONNX Runtime 的 `quantize_static()` 做 INT8 量化
5. 将 `.onnx` 模型文件打包到 HarmonyOS 应用的 `resources/rawfile/` 目录

### 6.4 推理流程在 ArkTS 中的实现

```
用户填写数据 → ArkTS 拼接 Prompt → Tokenizer（ArkTS 实现） → ONNX Runtime 推理 → 解码（ArkTS） → 生成文本 → 插入预览
```

**关键组件说明**：

| 组件 | 实现方式 | 说明 |
|------|---------|------|
| Tokenizer | ArkTS 实现简易版本（词表文件 + BPE 编码逻辑） | 鸿蒙无 Hugging Face Tokenizers 绑定，需手动实现 |
| ONNX Runtime | `@ohos.onnxruntime` (HarmonyOS NAPI 绑定) | 调用 `OrtCreateSession` 加载 `.onnx` 模型 |
| 推理引擎后端 | CPU (XNNPACK) / NPU (若支持) | INT8 量化模型优先使用 CPU XNNPACK 加速 |
| 解码策略 | Top-k (k=40) + Top-p (p=0.9) 采样 | 保证生成多样性同时避免乱码 |

### 6.5 性能目标估算

| 指标 | FP32 模型 | INT8 量化模型 |
|------|----------|--------------|
| 模型大小 | ~47MB | ~24MB |
| 推理时间 (Kirin 9000, CPU) | ~1.8s | ~0.9s |
| 推理时间 (Kirin 990, CPU) | ~2.5s | ~1.3s |
| 推理时间 (骁龙 8+, CPU) | ~1.6s | ~0.8s |
| 内存占用 (推理峰值) | ~280MB | ~160MB |
| 语法正确率 | ~98.5% | ~98.2% |
| 包体积影响 | ~50MB | ~27MB |

> ✅ INT8 量化模型在全部目标芯片上均可满足 ≤ 2s 的生成时间要求

## 八、附录

### 8.1 技术方案备忘

| 决策项 | 选择 | 理由 |
|--------|------|------|
| 模型框架 | PyTorch → ONNX | 鸿蒙端侧唯一可行的推理栈 |
| 量化方式 | INT8 静态量化 | 速度和精度的最佳平衡点 |
| Tokenizer 词表 | 20,000 tokens（中文 BPE） | 覆盖常见中文字词，控制嵌入层大小 |
| 最大序列长度 | 256 tokens | 一段经历描述通常 ≤ 200 字 |
| 本地存储 | `@ohos.data.preferences` + `@ohos.data.distributedKVStore` | 轻量级 KV 存储适合结构化数据 |
| PDF 生成 | `@ohos.hilog` + Canvas 绘制 PDF 或三方库转换 | 鸿蒙原生 PDF 生成能力有限，需自行排版绘制 |
| 状态管理 | `@State` + `@Prop` + `@Provide`/`@Consume` | ArkUI 原生状态管理，无需额外框架 |

### 8.2 鸿蒙端侧推理已知限制

1. ONNX Runtime for HarmonyOS 不支持 GPU/NPU 直接加速（仅 CPU）
2. 无 `transformers` 库，Tokenizer 需手写（可用 WASM 方式引入）
3. 模型热更新需通过应用市场整包更新（HSP 动态能力包可做增量）
4. 不支持动态 shape 输入，ONNX 需固定序列长度
